# Overview

If you're building an AI product, and you've got anything from a half-baked idea to a vibe-coded prototype to something already live, you're in the right place.

Building is faster than ever now. The hard part isn't shipping. It's figuring out what actually matters, and making sure you don't build yourself into a corner. The point of this manual is to help you think like a founder, and plant the seed for how founders make decisions.

AI has changed the way we build software. It's faster, cheaper, and more accessible than ever. So the hard part isn't execution anymore. It's judgment: picking the right problem, the right customer, and the right priorities before you build a bunch of the wrong things.

Here's what we'll cover:

- how software building is changing
- the difference between prototypes and real products
- what actually matters early
- how to avoid builder autopilot
- how to make something sustainable

The goal is simple: help you make the founder-level calls before you get too deep in the build.

## This manual is about the shift from builder to founder

| **A builder asks** | **A founder asks** |
| --- | --- |
| What should I make next? | Who is this for? |
| Which tool should I use? | What outcome matters most? |
| How fast can I ship this? | What creates leverage? |
| How do I add more features? | What makes someone stay? |
| Can I launch this? | Can this become a business? |

## Why it matters

Basically anyone can ship an MVP in a weekend now. But speed alone doesn't create traction, retention, or revenue. The founders who win won't be the ones shipping the fastest.

They'll be the ones making better decisions:

- choosing clearer problems
- understanding customers earlier
- focusing on retention before scale
- building systems instead of endless features

Execution got cheaper. Judgment got more valuable.

## Key outcomes

By the end of this manual, you should be able to:

- spot "default builder mode"
- make clearer product decisions
- focus on leverage instead of endless output
- think about retention, positioning, and distribution earlier
- think beyond launches and toward sustainability

## How to use this manual

<Callout>
If you want the full mindset shift, read it front to back. If you're short on time, start with **The shift in how software gets built** to get oriented. If you already have something live, jump to **Vibe coding vs real products**. Then use the later chapters as checkpoints while you build.
</Callout>

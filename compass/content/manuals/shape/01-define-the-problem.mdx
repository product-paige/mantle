Every good product starts the same way: someone had a problem that existing tools solved badly. Not a feature idea. Not a technology they wanted to use. A problem.

The reason this matters before anything else: you can't test demand for a solution. You can only test demand for a problem. If you can't describe the problem clearly, you don't know what you're building or who you're building it for — and everything downstream gets harder.

## Write the problem in one sentence

No features. No solution. Just the pain.

**Too vague:** "I want to build a project management tool for freelancers."

**Useful:** "Freelancers lose track of client deliverables when juggling more than three projects and miss deadlines."

The useful version names who feels the pain, what the pain is, and when it happens. If you can't write it that way, you're still guessing.

> If you can't write the problem in one sentence, you're not ready to build.

## The vitamin or painkiller test

Painkillers get bought. Vitamins get bookmarked.

The fastest test: are people already doing a workaround? Spreadsheets, duct-taped Zapier flows, manual processes they hate but can't stop doing. If yes, you might have a painkiller. If nobody's doing anything about the problem — it probably isn't painful enough.

## Checklist

- [ ] I can write the problem in one sentence: pain + who + when
- [ ] The sentence describes the problem, not a feature or solution
- [ ] I've identified at least one existing workaround people are using
- [ ] I can answer: is this a vitamin or a painkiller?

## Prompt

> *Help me write a clear problem statement for my startup idea.*
>
> *My rough idea is: [DESCRIBE IT]*
> *I think the target user is: [WHO YOU THINK IT'S FOR]*
>
> *Write three versions of the problem statement using this format: "[Specific who] struggle to [do what] when [context/trigger]." Go from vague to specific. Tell me which is strongest and why.*
